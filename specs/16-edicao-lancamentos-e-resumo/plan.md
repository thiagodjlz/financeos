# Plano de implementacao

## Abordagem

A feature nao exige mudanca de schema nem entidade nova: o backend de Lancamentos ja expoe `PUT /transactions/{id}` (`TransactionResource.update`, reaproveitando `apply()` compartilhado com o `POST`), entao o trabalho de edicao inline e 100% frontend — um segundo "modo" de formulario por linha da tabela (`editingId` + `editForm`, mesmo padrao ja usado em `features/users/users.ts` para alternar entre criar/editar, adaptado para ficar dentro da propria linha em vez de um painel separado) mais um metodo novo `update()` em `TransactionService`. Ja no Resumo, a exclusao de despesas `PENDING` dos totais e puramente uma mudanca de SQL em `DashboardRepository` (trocar a condicao `status <> 'CANCELED'` por `status = 'PAID'` nos tres metodos que somam despesa) e um ajuste de formula em `DashboardResource`; a reordenacao dos cards, o agrupamento de categorias em duas secoes e o reload automatico (trocando o botao "Atualizar" por `(change)` nos campos Ano/Mes) sao mudancas so de template/estilo em `dashboard.html`/`dashboard.ts`. A traducao de status para portugues fica centralizada num helper novo em `formatters.ts`, reaproveitado tanto na tabela (modo leitura) quanto — de forma implicita, via as mesmas opcoes ja usadas no formulario de criacao — no modo de edicao.

## Arquivos a alterar

### Backend

- `backend/src/main/java/br/com/financeos/dashboard/DashboardRepository.java`:
  - `totals()`: trocar a expressao do `total_expense` de `case when type = 'EXPENSE' and status <> 'CANCELED' then amount else 0 end` para `case when type = 'EXPENSE' and status = 'PAID' then amount else 0 end` (passa a ser numericamente igual a `paid_expense`, mas mantido como coluna separada por clareza e porque `DashboardSummaryResponse.totalExpense` continua existindo como campo proprio). `pending_expense` e `total_income` nao mudam.
  - `categoryBreakdown()`: trocar o filtro `where ... and (t.status is null or t.status <> 'CANCELED')` por uma condicao que discrimine por tipo: `and ((t.type = 'INCOME' and t.status is null) or (t.type = 'EXPENSE' and t.status = 'PAID'))` — mantem excluidas tanto despesas `CANCELED`/`PENDING` quanto receitas `CANCELED` (receita cancelada via `DELETE /transactions/{id}` grava `status = CANCELED` mesmo sendo `INCOME`, ver `TransactionResource.cancel()`, que nao diferencia tipo).
  - `monthlyEvolution()`: mesma troca do `total_expense` de `totals()`, aplicada a expressao de `expense` (`case when type = 'EXPENSE' and status <> 'CANCELED' then amount else 0 end` -> `... and status = 'PAID' ...`); `income` nao muda.
- `backend/src/main/java/br/com/financeos/dashboard/DashboardResource.java` — `summary()`: trocar o calculo do `balance` de `totals.totalIncome().subtract(totals.totalExpense())` para `totals.totalIncome().subtract(totals.paidExpense())` (equivalente ao novo `totalExpense` apos a mudanca acima, mas expresso de forma explicita conforme o criterio de aceite, e resiliente a qualquer divergencia futura entre os dois campos).
- `backend/src/test/java/br/com/financeos/dashboard/DashboardResourceTest.java` — `shouldReturnMonthlySummary()` cria um lancamento `EXPENSE/PENDING` de 300 em junho/2026 e hoje espera esse valor somado em `totalExpense`/`balance`/`categoryBreakdown`/`monthlyEvolution[5].expense`; apos a mudanca, os asserts precisam refletir que a despesa pendente some desses totais: `totalExpense` de `1500.00F` para `1200.00F`, `balance` de `3500.00F` para `3800.00F`, `categoryBreakdown` (tipo `EXPENSE`, "Sem categoria") de `1500.00F` para `1200.00F`, `monthlyEvolution[5].expense` de `1500.00F` para `1200.00F`; `pendingExpense` (`300.00F`) e `transactionCount` (`3`) continuam iguais.
- Nenhuma mudanca em `TransactionResource.java`, `FinancialTransaction.java`, `TransactionRequest.java`/`TransactionResponse.java` — `PUT /transactions/{id}` ja existe, ja chama `accessControl.require(Screen.TRANSACTIONS, Action.EDIT)` e ja reaproveita `apply()` (mesma logica de `status = null` para `INCOME` implementada na issue #14), entao nao ha ajuste de backend para a edicao inline em si.

### Frontend

- `frontend/src/app/core/formatters.ts` — adicionar `transactionStatusLabel(status: TransactionStatus | null): string`, mapeando `PENDING -> 'Pendente'`, `PAID -> 'Pago'`, `CANCELED -> 'Cancelado'`, `null -> '-'` (usado hoje so por receitas, que nao tem status). Importa `TransactionStatus` de `../models`.
- `frontend/src/app/core/services/transaction.service.ts` — adicionar `update(id: string, payload: Partial<Transaction>): Promise<Transaction>` fazendo `PUT ${API_BASE}/transactions/${id}`, no mesmo padrao de `create()` (que ja existe) e de `UserService.update()` (`frontend/src/app/core/services/user.service.ts`).
- `frontend/src/app/features/transactions/transactions.ts`:
  - novos signals: `editingId = signal<string | null>(null)` (id do lancamento em edicao, `null` = nenhuma linha em edicao), `editCategories = signal<Category[]>([])` (categorias filtradas para o dropdown da linha em edicao — sinal separado de `filteredCategories`, que continua servindo so ao formulario de criacao), `confirmingExit = signal(false)` (controla a exibicao inline da pergunta "Deseja sair sem salvar?" na linha em edicao).
  - novo campo `editForm` (mesma forma de `transactionForm`: `transactionDate, description, amount, type, status, categoryId`) e um snapshot privado `editSnapshot` (copia congelada dos valores originais do lancamento ao entrar em edicao) usado para o diff de "alteracao pendente".
  - `protected startEdit(transaction: Transaction): void` — no-op se `editingId()` ja estiver setado (garante uma linha por vez, alem do `[disabled]` no botao "Editar" das outras linhas); preenche `editForm` a partir de `transaction`, guarda `editSnapshot = { ...editForm }`, chama `loadCategoriesForEdit(transaction.type)`, zera `confirmingExit`.
  - `protected async onEditTypeChange(): Promise<void>` — mesmo comportamento de `onTypeChange()` (ja existente para o formulario de criacao) mas usando `editForm`/`editCategories`: recarrega `editCategories` via `categoryService.listByType(editForm.type)` e limpa `editForm.categoryId` se a categoria atual nao pertencer ao novo tipo.
  - `private async loadCategoriesForEdit(type: TransactionType): Promise<void>` — `editCategories.set(await this.categoryService.listByType(type))` (espelha `loadCategoriesForType`, sem mexer no signal `filteredCategories` do formulario de criacao).
  - `protected isEditDirty(): boolean` — compara `editForm` com `editSnapshot` campo a campo (ou `JSON.stringify`), usado tanto por `requestExit()` quanto, potencialmente, para desabilitar "Salvar" se nada mudou (nao exigido pela spec, mas nao ha conflito).
  - `protected async saveEdit(transaction: Transaction): Promise<void>` — monta o payload igual a `saveTransaction()` (`amount: Number(...)`, `status: editForm.type === 'INCOME' ? null : editForm.status`, `categoryId: emptyToNull(editForm.categoryId)`), chama `transactionService.update(transaction.id, payload)`, depois `await transactionService.refresh()` (repopula a tabela a partir do servidor, cobrindo o criterio "atualiza a linha na tabela com os novos valores"), e por fim `editingId.set(null)`; usa o signal `saving` ja existente e trata erro com `error.set(...)`, no mesmo padrao de `saveTransaction()`/`cancelTransaction()`.
  - `protected requestExit(): void` — se `!isEditDirty()`, chama `exitEditDiscarding()` direto; senao, `confirmingExit.set(true)`.
  - `protected async confirmExitYes(): Promise<void>` — `await transactionService.refresh()` (recarrega do servidor, descartando qualquer alteracao local), depois `exitEditDiscarding()`.
  - `protected confirmExitNo(): void` — `confirmingExit.set(false)` (mantem `editingId` e `editForm` como estao, preservando os valores alterados).
  - `private exitEditDiscarding(): void` — `editingId.set(null); confirmingExit.set(false);`.
  - `protected statusLabel(status: TransactionStatus | null): string` — delegate fino para `transactionStatusLabel` (import de `../../core/formatters`), usado no template.

## Riscos e pontos de atencao

- **Editar um lancamento ja `CANCELED` (resolvido)** — decisao tomada com o usuario: permitir editar normalmente e reativar. O botao "Editar" nao tem guarda contra `transaction.status === 'CANCELED'`, e o dropdown de Status na linha de edicao (como no formulario de criacao) so oferece "Pendente"/"Pago"; ao salvar, o lancamento sai do estado `CANCELED` e passa a ter o status escolhido — comportamento intencional, sem tratamento especial. Ver criterio de aceite correspondente em `spec.md`.
- **`categoryBreakdown` numericamente igual a `paidExpense`/`totalExpense` apos a mudanca** — o campo `totalExpense` no card "Despesas" passa a ter o mesmo valor de `paidExpense` no card "Pendente" oculto... na pratica os dois campos (`totalExpense`, `paidExpense`) ficam redundantes no payload da API (ambos so somam `PAID`), mas ambos continuam expostos no `DashboardSummaryResponse` porque o card "Pendente" usa `pendingExpense` (esse sim continua diferente) — nao remover nenhum campo do record, so mudar o SQL, para nao quebrar o contrato da API sem necessidade.
- **`(change)` vs `(ngModelChange)` no campo Ano** — usar `(ngModelChange)` dispararia uma requisicao a cada digito digitado (ano tem 4 digitos, entao ate 4 chamadas por edicao, algumas com anos invalidos/incompletos tipo "2" ou "20"); o plano usa `(change)` (evento nativo, dispara so ao sair do campo/perder foco) especificamente para evitar isso — nao trocar por `(ngModelChange)` durante a implementacao achando que e equivalente.
- **`editingId`/`filteredCategories` vs `editCategories`** — nao reaproveitar o signal `filteredCategories` (usado pelo formulario de criacao) para a linha em edicao; sao fluxos independentes (um usuario pode estar preenchendo o formulario de criacao com Tipo = Receita ao mesmo tempo em que edita uma linha de Despesa) e compartilhar o signal misturaria os dropdowns.
- **`name` duplicado em inputs do Angular Forms** — a pagina tem, ao mesmo tempo, o `<form>` de criacao (com `name="transactionDate"`, `name="type"` etc.) e, quando uma linha entra em edicao, os inputs da linha de edicao; usar nomes diferentes (`editDate`, `editType`, ...) evita colisao de `NgModel`/`NgForm`, ja que ambos os grupos de campos convivem no mesmo template.
- **Teste de backend existente (`DashboardResourceTest`) vai falhar sem atualizacao** — os valores esperados hoje (`totalExpense: 1500.00F`, `balance: 3500.00F` etc.) assumem que `PENDING` conta nos totais; a mudanca de SQL descrita neste plano so fica "verde" se o teste for atualizado junto (passo 3 da sequencia), nao depois.
- **`knowledge/transactions.md`** — confirma que `DELETE /transactions/{id}` (Cancelar) so muda `status` para `CANCELED`, sem hard delete, e que nao ha validacao cruzada de `categoryId`/`type`; nenhuma dessas regras muda nesta issue, mas a UI de edicao inline e a primeira a expor esses dois fatos diretamente ao usuario (categoria "errada" para o tipo so e evitada no frontend, nunca no backend).
